# HW10 · 5 ключових понять простими словами

Це 5 речей, які варто винести з цієї домашки. Якщо розумієш ці п'ять — розумієш, чим production RAG API відрізняється від «hello world з LangChain».

---

## 1. RAG — пошук плюс генерація (Retrieval-Augmented Generation)

**Простими словами**: ти не питаєш модель «з пам'яті», а спершу шукаєш у своєму документі шматок із відповіддю, кладеш його в prompt як контекст — і модель відповідає **на основі цього шматка**.

**Навіщо**: LLM нічого не знає про твій документ (PDF, README, внутрішня вікі). Без RAG вона або вигадає («галюцинація» — hallucination), або скаже «не знаю». З RAG ти контролюєш, з яких джерел вона бере факти.

**Як це працює всередині**:
1. **Індексація** (одноразово, у `scripts/index.py`): документ ріжеш на шматки (chunks) по ~500 токенів, кожен переганяєш через модель ембедингів (embedding model) в число-вектор (384 числа для `all-MiniLM-L6-v2`), складаєш у векторну базу (vector DB).
2. **На запит** (на кожен `/chat/stream`): запит користувача теж переганяєш у такий самий 384-вимірний вектор, шукаєш у базі 3 найближчі chunk-и за косинусною відстанню (cosine similarity), вставляєш їхній текст у prompt як контекст.

**Чому ембединг — це «зміст», а не «слова»**: модель ембедингів навчена так, що «кіт» і «кошеня» мають близькі вектори, а «кіт» і «бухгалтерія» — далекі. Тому пошук знаходить **за змістом**, навіть якщо слів буквально немає в документі.

**Гачок** (gotcha): індекс і запит **обов'язково** мають проходити через **ту саму модель ембедингів**. Інакше вектори несумісні — як міряти кілограми лінійкою.

## 2. SSE Streaming — токени летять по одному

**Простими словами**: замість того щоб чекати 5 секунд на повну відповідь модельки і отримати її блоком, ти отримуєш слово за словом — як у ChatGPT, де текст друкується. Це Server-Sent Events (SSE).

**Навіщо**: TTFT (Time To First Token) = 200мс замість 5000мс на повну відповідь. UX-сприйняття швидкості зростає в рази, навіть якщо повна генерація триває стільки ж.

**Як виглядає на дроті** (на провід, on the wire):
```
data: {"type":"token","content":"Це"}

data: {"type":"token","content":" дванадцять"}

data: {"type":"done","sources":["chunk_12"],"usage":{...}}
```
Звичайний HTTP-response, але сервер **не закриває з'єднання**, а пише шматки в міру їх готовності. Клієнт читає через `EventSource` (браузер) або `curl -N` (термінал, прапорець `-N` = no buffering).

**FastAPI рецепт**: `StreamingResponse(async_generator(), media_type="text/event-stream")`. У генераторі — `yield "data: {...}\n\n"` на кожен токен. Два `\n` обов'язкові — це роздільник SSE-події.

**Disconnect handling**: користувач закрив вкладку → треба зупинити LLM-запит, бо платиш за токени, які ніхто не побачить. У циклі генерації перевіряєш `await request.is_disconnected()` — якщо так, кидаєш `CancelledError`, який пробивається в OpenRouter SDK і скасовує запит.

**Гачок**: nginx/Cloudflare за замовчуванням буферизують response. Треба слати заголовок `X-Accel-Buffering: no` і вимикати буферизацію на проксі — інакше клієнт побачить порожньо 5 секунд, потім всю відповідь блоком.

## 3. Semantic Cache — кеш «за змістом», а не за рядком

**Простими словами**: звичайний кеш (Redis `GET/SET`) працює тільки якщо ключ збігається **символ у символ**. Питання «Що таке config?» і «What is configuration?» для нього — різні ключі, тому кеш не спрацює. Semantic cache знаходить, що ці питання мають **схожий зміст**, і повертає закешовану відповідь.

**Як влаштовано**:
1. На запит → береш ембединг запиту (той самий, що йде в RAG retrieval — **один виклик**, не два).
2. Шукаєш у `cache_collection` найближчий вектор. Якщо `similarity > 0.92` → це той самий зміст, повертаєш збережену відповідь з payload.
3. Якщо схожого немає (MISS) → робиш LLM-виклик, після стріму **зберігаєш** `(embedding, query, response, expire_at=now+3600)` у ту саму колекцію.

**Чому threshold = 0.92**: емпірична планка. Нижче 0.85 — починає віддавати «схожі за темою, але різні питання» (false positives, помилкові потрапляння). Вище 0.95 — кеш майже не спрацьовує. 0.92 — типовий sweet spot для MiniLM.

**Економіка**: cache HIT коштує ~0мс LLM-часу і $0. Cache MISS — повний LLM call (~2 секунди, ~$0.001). При cache hit rate 40% витрати падають майже навпіл. Це і є головна причина, чому в production додають semantic cache: **не за швидкість, а за гроші**.

**Гачок з TTL**: Qdrant не має вбудованого TTL (на відміну від Redis). Тому термін придатності зберігаєш у payload (`expire_at` timestamp) і фільтруєш на read або чистиш окремою задачею. Кеш «отруюється» — без TTL застаріла інформація живе вічно.

**UX-деталь**: на HIT теж стрімиш токени по одному (з невеликою затримкою), щоб користувач не бачив несподівано миттєвий response — інакше відчуття «глюк, відповідь з'явилась миттєво» зламає очікування.

## 4. Token-based Rate Limiting — справедливість «за газом», а не «за кліками»

**Простими словами**: класичний rate limit рахує **кількість запитів** («100 запитів за хвилину»). Але для LLM один запит може коштувати $0.0001, а інший — $0.50 (велика модель + великий контекст). Тому справедливо лічити **токени** (вхід + вихід), а не запити.

**Як працює (token bucket)**:
- У кожного API-ключа є «відерце» (bucket) розміром, наприклад, 20 000 токенів.
- Кожен запит витрачає реально спалені токени (input + output, дізнаєшся з `usage` від OpenRouter після стріму).
- Відерце поповнюється рівномірно: 20K / 60с = ~333 токени на секунду.
- Якщо відерце порожнє → 429 + заголовок `Retry-After: 23` (через стільки секунд буде достатньо токенів для типового запиту).

**Чому в Redis**: rate limit має бути спільним між інстансами сервісу. Якщо в тебе 3 машини за load-balancer-ом, локальний counter не зупинить зловмисника, який балансує запити по машинах. Redis — централізований counter, atomic INCR гарантує, що навіть при race condition підрахунок коректний.

**Хитрість з Upstash**: серйозні реалізації token bucket пишуть на Lua-скриптах для атомарності (Lua виконується в Redis як одна операція). Upstash REST API не підтримує Lua → користуєшся комбінацією `INCR + EXPIRE` зі знанням, що між цими двома командами теоретично існує мікросекундне вікно «гонки» (race). Для домашки прийнятно; для платіжного gateway — взяв би Redis Cloud з RESP-протоколом.

**Передоплата vs пост-оплата**: ідеально лічити токени **після** LLM-відповіді (бо лиш тоді знаєш точну кількість output-токенів). Але тоді користувач може перевищити ліміт на одну хвилину. Компроміс: робиш попередню оцінку (tiktoken на вхід + 500 токенів на можливий вихід), перевіряєш — і коригуєш по факту після стріму.

## 5. Multi-Provider Fallback + Circuit Breaker — «жоден провайдер не безсмертний»

**Простими словами**: будь-який AI-провайдер періодично падає (`502`), тротлить (`429`), або тупо повільний. Production-сервіс не може дозволити собі лягати разом із ним — потрібен план «б», «в», «г». Це **fallback chain**.

**Як влаштовано**:
1. У конфізі для кожного tier є **список з 3 моделей**:
   ```
   [openai/gpt-4o-mini,           # primary
    mistralai/mistral-small,      # fallback 1 — інший провайдер
    meta-llama/llama-3.2-3b:free] # fallback 2 — крайній варіант, безкоштовний
   ```
2. Намагаєшся primary з timeout 15 секунд (`asyncio.wait_for`).
3. Помилка типу «можна повторити» (retryable: `429, 500, 502, 503, 504, TimeoutError`) → переходиш на наступну модель у списку, прапорець `fallback_used=True`.
4. Помилка типу «не повторювати» (non-retryable: `400` погані параметри, `401` auth, `403` доступ, `422` validation, content filter) → одразу повертаєш помилку клієнту. Сенсу пробувати іншу модель немає — те саме отримаєш.

**Чому OpenRouter спрощує це**: один API-ключ, один SDK (сумісний з OpenAI Python SDK), 200+ моделей. Без OpenRouter довелось би тримати ключі від OpenAI, Anthropic, Mistral, Google окремо — і знати специфіку кожного SDK.

**Circuit Breaker — стан «не ламай те, що вже зламане»**:
- Якщо primary дала **5+ помилок за 60 секунд** → відкриваєш «запобіжник» (circuit open).
- Поки запобіжник відкритий (60 секунд) → не пробуєш primary взагалі, одразу йдеш на fallback.
- Через 60 секунд → пробний запит. Успіх → запобіжник закривається.
- Сенс: коли провайдер «лежить», не додавай йому навантаження своїми retry-ями (у нього й так лавина). Дай оклематись.

**Стани автомата** (state machine):
```
  CLOSED ──5 fails──► OPEN ──60s──► HALF_OPEN ──success──► CLOSED
                                          └──fail──► OPEN
```

**Що логуємо**: `fallback_used: True` у cost record і `model: <реально_використана>` (не та, що в primary конфізі). Потім `/usage/breakdown` показує `fallback_rate` — якщо це 50%, значить primary лежить наполовину, треба міняти.

## Як ці 5 з'єднані в одному `/chat/stream`

```
запит → auth(API key)
      → rate_limit(token bucket в Redis)        ← (4)
      → security(prompt injection check)
      → embed(query) — один раз!
      → cache.lookup(embedding, threshold=0.92)  ← (3)
          ├─ HIT  → стрім закешованої відповіді → SSE done(cache_hit=True)
          └─ MISS → retriever.search(embedding, k=3) ← (1) RAG
                  → llm_router.stream_with_fallback(  ← (5) fallback chain
                      tier_models, prompt+chunks)
                  → SSE stream токени по одному    ← (2) SSE
                  → cache.store(embedding, response, ttl=1h)
                  → cost_tracker.log(...)
                  → SSE done(sources, cost, fallback_used)
```

Те, що раніше робив LangChain під капотом «одним рядком», тепер ти бачиш руками — і розумієш, де можна поламатись, де можна сплатити зайве, де можна злити дані. Це і є різниця між «зробив демо за гайдом» і «можу підтримувати production».
