Інтерактивна візуалізація архітектури трансформера

0. Конфігурація моделі

Ці параметри визначають розмір і поведінку моделі. Змінюй слайдери, щоб побачити наслідки:

Що відбувається при зміні параметрів:

Більше N_EMBED / N_HEAD / N_LAYER = модель розумніша, але:
— більше пам'яті (може не влізти в GPU)
— повільніше тренування і генерація
— потребує більше даних (інакше overfitting)

BLOCK_SIZE = скільки символів модель "бачить" одночасно. 128 = модель не знає що було 129 символів тому.
GPT-4 має контекст ~128,000 токенів. Наша модель — 128 символів.

DROPOUT = при тренуванні випадково "вимикає" частину нейронів. Захист від зазубрювання.
0.0 = немає захисту (overfitting). 0.5 = занадто агресивно (модель не вчиться).

LR (Learning Rate) = наскільки сильно оновлювати ваги на кожному кроці.
Занадто великий = тренування "стрибає" і не збігається. Занадто маленький = вчиться тижнями.
Конфігураціяnano_gpt.py
BLOCK_SIZE = 128    # довжина контексту
BATCH_SIZE = 64     # розмір батчу
N_EMBED    = 192    # розмірність вектора
N_HEAD     = 6      # кількість attention-голів
N_LAYER    = 4      # кількість transformer-блоків
DROPOUT    = 0.2    # регуляризація
LR         = 3e-4   # learning rate
MAX_ITERS  = 1500   # кроків тренування

1. Токенізація

↓
Що відбувається: Модель не розуміє букви — вона працює з числами. Кожен символ отримує унікальний індекс зі словника (vocabulary).

На прикладі "hello": кожна буква стає числом — h→7, e→4, l→11, l→11, o→14. Зверни увагу: обидві "l" отримують однаковий індекс (11) — модель поки не знає що вони на різних позиціях. Це виправиться на наступному кроці.

Кожна буква = окремий токен. Це означає, що слово "hello" — це 5 незалежних одиниць, кожна проходить через весь transformer окремо (свій embedding, свій Q/K/V, свій MLP). Вони "спілкуються" тільки в одному місці — Self-Attention.
Порівняння: наша nano-GPT vs реальний GPT-4
nano-GPT (char-level) GPT-4 (BPE)
"hello" h e l l o = 5 токенів hello = 1 токен
"hello world" h e l l o ␣ w o r l d = 11 токенів hello ␣world = 2 токени
"привіт" п р и в і т = 6 токенів при ві т = 3 токени
Словник ~65 символів ~100,000 subwords
Обчислень на "hello world" 11 × весь pipeline 2 × весь pipeline
Висновок: BPE ефективніший — менше токенів = менше обчислень, кожен токен несе більше сенсу. Саме тому API рахує ціну в токенах, а не в словах чи символах. Наша char-level модель простіша для розуміння — але принцип роботи той самий.
Токенізаціяnano_gpt.py
chars = sorted(set(TEXT))                       # унікальні символи
vocab_size = len(chars)                          # розмір словника
stoi = {c: i for i, c in enumerate(chars)}       # символ → число
itos = {i: c for i, c in enumerate(chars)}       # число → символ
encode = lambda s: [stoi[c] for c in s]          # "hello" → [7, 4, 11, 11, 14]
decode = lambda ids: "".join(itos[i] for i in ids)

data = torch.tensor(encode(TEXT), dtype=torch.long)
train_data, val_data = data[:int(0.9*len(data))], data[int(0.9*len(data)):]

def get_batch(split):
    d = train_data if split == "train" else val_data
    ix = torch.randint(len(d) - BLOCK_SIZE - 1, (BATCH_SIZE,))
    x = torch.stack([d[i : i+BLOCK_SIZE] for i in ix])       # вхід
    y = torch.stack([d[i+1 : i+1+BLOCK_SIZE] for i in ix])   # ціль (зсув на 1)
    return x.to(device), y.to(device)

2. Embedding = Token + Position

Token Embedding — хто цей символ?
+
Position Embedding — де він стоїть?
=
Combined Embedding — вхід у трансформер
Що відбувається: Кожен токен (число) перетворюється на вектор із 192 чисел — це його "координати" в просторі значень.

На прикладі "hello":
— "h" (ID 7) → шукаємо 7-й рядок у таблиці token embedding → вектор [0.12, -0.34, ...] (192 числа)
— Додаємо position embedding для позиції 0 → вектор [0.01, 0.05, ...]
— Результат: [0.13, -0.29, ...] — тепер модель знає і ХТО цей символ, і ДЕ він стоїть
— Дві "l" мають однаковий token embedding, але різний position embedding — тому вони різні!

Навіщо позиція? Self-attention — це операція над множиною, вона не розрізняє порядок. Без position embedding "hello" і "olleh" виглядають однаково.

Чому сума, а не конкатенація? Конкатенація подвоїла б розмірність (384 замість 192). Сума працює, бо модель вчиться кодувати ідентичність і позицію в ортогональних підпросторах.
Embeddings (частина NanoGPT)nano_gpt.py
# В __init__:

self.pos_emb = nn.Embedding(BLOCK_SIZE, N_EMBED)  # позиція → вектор 192d

# В forward:

3. Self-Attention

Клікни на токен, щоб побачити як він рахує attention:

На прикладі "hello" — що робить токен "o" (остання позиція):
1. "o" генерує свій Q (Query): "я — буква 'o' в кінці слова, що мені потрібно?"
2. Кожен токен генерує K (Key): h каже "я — початок слова", l каже "я — повтор"
3. "o" рахує Q·K з кожним: h→5%, e→8%, l→3%, l→6%, o→78% (сам на себе — найбільше)
4. Causal mask: "o" бачить h, e, l, l, o — але НЕ бачить нічого після себе
5. Зважена сума V: output = 5%·Vh + 8%·Ve + 3%·Vl + 6%·Vl + 78%·Vo
6. Тепер "o" знає: "я останній у слові 'hello'" — це допоможе передбачити пробіл

Алгоритм (формально): Q·KT/√dim → mask → softmax → ·V

Чому O(n²)? Кожен токен рахує скалярний добуток з КОЖНИМ попереднім. 5 токенів = 15 добутків. 100k токенів = 10 мільярдів. Тому довгий контекст — дорогий.
Self-Attentionnano_gpt.py
class CausalSelfAttention(nn.Module):
    def __init__(self):
        super().__init__()
        self.qkv = nn.Linear(N_EMBED, 3 * N_EMBED)   # Q, K, V одним множенням
        self.proj = nn.Linear(N_EMBED, N_EMBED)        # проєкція після concat голів
        self.drop = nn.Dropout(DROPOUT)
        self.register_buffer("mask",                    # causal mask
            torch.tril(torch.ones(BLOCK_SIZE, BLOCK_SIZE))
            .view(1, 1, BLOCK_SIZE, BLOCK_SIZE))

    def forward(self, x):
        B, T, C = x.shape
        q, k, v = self.qkv(x).split(N_EMBED, dim=2)          # (B, T, 192) кожен

        head_dim = C // N_HEAD                                  # 32
        q = q.view(B, T, N_HEAD, head_dim).transpose(1, 2)    # (B, 6, T, 32)
        k = k.view(B, T, N_HEAD, head_dim).transpose(1, 2)
        v = v.view(B, T, N_HEAD, head_dim).transpose(1, 2)

        att = (q @ k.transpose(-2, -1)) / math.sqrt(head_dim) # (B, 6, T, T)
        att = att.masked_fill(self.mask[:,:,:T,:T] == 0, float("-inf"))
        att = F.softmax(att, dim=-1)
        att = self.drop(att)

        y = att @ v                                             # (B, 6, T, 32)
        y = y.transpose(1, 2).contiguous().view(B, T, C)      # (B, T, 192)
        return self.proj(y)

4. Multi-Head Attention

6 голів дивляться на текст ПАРАЛЕЛЬНО — кожна бачить "своє":

На прикладі "hello":
— Голова 1 (позиційна): "o" дивиться на "l" (попередній символ) — слідкує за порядком
— Голова 2 (початок): "o" дивиться на "h" — пам'ятає початок слова
— Голова 5 (схожість): "o" дивиться на "o" — знаходить повтори
— Кожна голова бачить щось інше в тих самих 5 токенах!

Як це працює з розмірностями:
— Embedding токена: 192 dim
— Ділимо на 6 голів: кожна працює з 192/6 = 32 dim
— Кожна голова рахує свій attention (Q·K→softmax→·V) на 32-dim векторах
— Результати 6 голів конкатенуються: 6 × 32 = 192 dim (повернулись до оригіналу!)
— Фінальна проєкція Wout змішує інформацію від усіх голів

Аналогія: 6 людей читають "hello" — один дивиться на граматику, інший на позицію букв, третій на повтори. Потім вони обговорюють і дають спільний висновок.

# Розбиваємо 192 dim на 6 голів по 32 dim
head_dim = C // N_HEAD                                  # 192 / 6 = 32
q = q.view(B, T, N_HEAD, head_dim).transpose(1, 2)    # (B, T, 192) → (B, 6, T, 32)
k = k.view(B, T, N_HEAD, head_dim).transpose(1, 2)
v = v.view(B, T, N_HEAD, head_dim).transpose(1, 2)

# ... кожна голова рахує свій attention ...

# Збираємо 6 голів назад
y = y.transpose(1, 2).contiguous().view(B, T, C)      # (B, 6, T, 32) → (B, T, 192)
return self.proj(y)                                     # проєкція змішує голови

5. Causal Mask

На прикладі "hello":
— "h" (позиція 0): бачить тільки себе → ✓✗✗✗✗
— "e" (позиція 1): бачить h і себе → ✓✓✗✗✗
— "o" (позиція 4): бачить h, e, l, l, o → ✓✓✓✓✓
— Ніхто не бачить символи ПІСЛЯ себе — це і є "causal"

Чому це критично? GPT навчений передбачати НАСТУПНИЙ символ, не знаючи що далі. Якщо "o" побачить що після нього пробіл — навіщо йому вчитись? Він просто скопіює відповідь. Маска змушує модель ДУМАТИ, а не списувати.

GPT vs BERT: GPT використовує causal (трикутну) маску — бачить тільки минуле. BERT бачить ВСЕ речення в обидва боки — тому він кращий для розуміння, але не вміє генерувати.

# В forward:

6. LayerNorm

LayerNorm
нормалізація значень
Що робить LayerNorm
Бере вектор токена (192 числа) і нормалізує: mean = 0, std = 1.
Формула: output = (x - mean) / std * gamma + beta
де gamma і beta — навчальні параметри (модель вчиться "де нормалізувати вільніше").
На прикладі "hello" — вектор токена "o"
До LayerNorm: [12.5, -3.2, 0.8, 45.1, -22.0, 1.1, ...] ← числа "розкидані"
Після LayerNorm: [0.31, -0.15, 0.01, 1.42, -0.87, 0.02, ...] ← стабільний діапазон
Навіщо? Без нормалізації значення після кожного блоку ростуть або зменшуються. Через 4 блоки числа можуть стати 10,000 або 0.0001 — softmax і градієнти ламаються.

Де стоїть? Перед attention і перед MLP (pre-norm). Це стабілізує тренування.

В коді: self.ln1 = nn.LayerNorm(N_EMBED) — один рядок. Але без нього модель глибше 2-3 шарів не тренується.

Аналогія: після кожної наради (attention) і кожного обдумування (MLP) — "перезавантаження", щоб наступний раунд починав з чистого стану, а не з накопиченого шуму.
LayerNormnano_gpt.py → Block
# В __init__:
self.ln1 = nn.LayerNorm(N_EMBED)    # 192 параметри gamma + 192 beta
self.ln2 = nn.LayerNorm(N_EMBED)

# В forward (pre-norm — нормалізуємо ПЕРЕД attention і MLP):
x = x + self.attn(self.ln1(x))      # LayerNorm → Attention
x = x + self.mlp(self.ln2(x))       # LayerNorm → MLP

7. Residual Connection

Input x
→
Attention(x)
→
x + Attention(x)
На прикладі "hello" — токен "o"
x = [0.31, -0.15, 0.01, 1.42, ...] ← оригінал
attn(x) = [0.08, 0.22, -0.05, 0.15, ...] ← що дізнався від інших токенів
x + attn = [0.39, 0.07, -0.04, 1.57, ...] ← знає і хто він, і що дізнався
Навіщо додавати оригінал? Без residual модель "забуває" що знала до attention. Attention каже "ти схожий на 'h' і 'e'" — але без оригіналу модель забуде що вона — "o".

Проблема градієнтів: у глибокій мережі (4+ шари) градієнти при backpropagation множаться багато разів і зникають (vanishing gradients). Residual створює "шосе" — градієнт може пройти через + напряму, минаючи всі шари.

В коді: x = x + self.attn(self.ln1(x)) — один знак "+", але без нього мережа не тренується.

Факт: ResNet (2015) показав що residual connections дозволяють тренувати мережі зі 152 шарами замість 20. Трансформер використовує ту саму ідею.
Residual Connectionnano_gpt.py → Block.forward()
# Residual = один символ "+":
x = x + self.attn(self.ln1(x))   # x (оригінал) + результат attention
x = x + self.mlp(self.ln2(x))    # x (попередній) + результат MLP

# Без "+" (без residual) — модель глибше 4 шарів не тренується
# x = self.attn(self.ln1(x))     # ← так НЕ працює

8. MLP (Feed-Forward)

Що таке MLP?
MLP (Multi-Layer Perceptron) — проста нейронна мережа з двох лінійних шарів з нелінійністю між ними. Це найпростіший "мозок" нейромережі.

У трансформері MLP стоїть після attention в кожному блоці:
— Attention = "нарада" — токени обмінялись інформацією
— MLP = "подумати самому" — кожен токен обробляє те, що дізнався. Без обміну з іншими.

Факт: ~2/3 всіх параметрів трансформера знаходяться саме в MLP, не в Attention.

На прикладі "hello" — токен "o" після attention
Вхід (192 dim): [0.39, 0.07, -0.04, 1.57, ...]
Після expand (768 dim): [0.12, -0.45, 0.88, 0.03, -0.21, 1.34, 0.55, ...] ← в 4 рази більше місця
Після GELU: [0.10, 0.00, 0.76, 0.02, 0.00, 1.22, 0.43, ...] ← негативні ≈ 0
Після compress (192 dim): [0.52, -0.11, 0.33, 0.89, ...] ← назад до 192
Навіщо розширення 192 → 768? У 192 вимірах "тісно" для складних обчислень. Розширюємо в 4 рази — створюємо "простір для думання". Після обробки стискаємо назад, залишаючи тільки корисну інформацію.

GELU — навіщо нелінійність?
Без GELU два лінійних шари підряд математично = один лінійний шар. W2·(W1·x) = W3·x — скільки шарів не став, модель вчить тільки прямі лінії.
GELU "згинає" простір між ними — модель може вчити криві, пороги, складні патерни.

Що GELU робить з числами: м'який порог — позитивні значення проходять, негативні гасяться.
2.0→1.96 (пропускає)   0.5→0.35 (трохи гасить)   -0.5→-0.15 (сильно гасить)   -2.0→-0.05 (≈нуль)

Навіщо саме в MLP: після розширення до 768 dim, модель вчиться детектувати "фічі" — наприклад фіча 42 = "кінець слова" (позитивне → пропускає), фіча 200 = "початок речення" (негативне → гасить). GELU працює як фільтр: активує релевантне, гасить нерелевантне.

В коді:
nn.Linear(192, 768) — розширення
nn.GELU() — нелінійність (фільтр фіч)
nn.Linear(768, 192) — стиснення

Факт: ~2/3 всіх параметрів трансформера знаходяться в MLP, не в Attention.
MLP (Feed-Forward)nano_gpt.py → Block
self.mlp = nn.Sequential(
    nn.Linear(N_EMBED, 4 * N_EMBED),   # 192 → 768 (розширення)
    nn.GELU(),                           # нелінійність (фільтр фіч)
    nn.Linear(4 * N_EMBED, N_EMBED),    # 768 → 192 (стиснення)
    nn.Dropout(DROPOUT),                 # регуляризація
)

9. How It Works Together

Цей блок повторюється ×4 (N_LAYER = 4). Вихід одного блоку → вхід наступного. Однаковий shape: (batch, seq_len, 192)
На прикладі "hello":
— Блок 1: токен "o" дізнається що він у слові (attention до h,e,l,l). MLP обробляє: "це кінець слова"
— Блок 2: глибше — "це кінець слова 'hello', яке часто зустрічається перед пробілом"
— Блок 3: ще глибше — "після 'hello' зазвичай йде ' world' або ' there'"
— Блок 4: фінальне рішення: вектор "o" тепер кодує "наступний символ — пробіл (41%)"

Attention змішує інформацію МІЖ токенами — "нарада з колегами".
MLP обробляє кожен токен ОКРЕМО (192→768→192) — "подумати самому".
Residual (+) — "не забути що знав до наради". Без них глибша за 4 шари мережа не тренується.
LayerNorm — стабілізація значень. Без нього числа ростуть/зменшуються з кожним блоком.

Масштаб: Наша модель — 4 блоки. GPT-2 — 12. GPT-3 — 96. GPT-4 — ймовірно 120+. Більше блоків = більше "раундів роздумів".
Повна модель NanoGPTnano_gpt.py
class Block(nn.Module):
    def __init__(self):
        super().__init__()
        self.ln1  = nn.LayerNorm(N_EMBED)
        self.attn = CausalSelfAttention()
        self.ln2  = nn.LayerNorm(N_EMBED)
        self.mlp  = nn.Sequential(
            nn.Linear(N_EMBED, 4 * N_EMBED), nn.GELU(),
            nn.Linear(4 * N_EMBED, N_EMBED), nn.Dropout(DROPOUT))

    def forward(self, x):
        x = x + self.attn(self.ln1(x))   # attention + residual
        x = x + self.mlp(self.ln2(x))    # MLP + residual
        return x

class NanoGPT(nn.Module):
    def __init__(self):
        super().__init__()
        self.tok_emb = nn.Embedding(vocab_size, N_EMBED)
        self.pos_emb = nn.Embedding(BLOCK_SIZE, N_EMBED)
        self.blocks  = nn.Sequential(*[Block() for _ in range(N_LAYER)])
        self.ln_f    = nn.LayerNorm(N_EMBED)
        self.head    = nn.Linear(N_EMBED, vocab_size, bias=False)

    def forward(self, idx, targets=None):
        B, T = idx.shape
        x = self.tok_emb(idx) + self.pos_emb(torch.arange(T, device=idx.device))
        x = self.ln_f(self.blocks(x))
        logits = self.head(x)
        loss = None
        if targets is not None:
            loss = F.cross_entropy(logits.view(-1, vocab_size), targets.view(-1))
        return logits, loss

На прикладі "hello" → " world":
Крок 1: "hello" → весь pipeline → ␣:41% → обираємо пробіл
Крок 2: "hello " → весь pipeline → w:38% → обираємо "w"
Крок 3: "hello w" → весь pipeline → o:45% → обираємо "o"
...і так далі поки не вийде "hello world!"

Зверни увагу: з кожним кроком модель стає впевненішою (41% → 45% → 52% → 73%). Це тому, що контекст "hello wor" сильно звужує варіанти — мало слів починаються на "wor".

Temperature керує "впевненістю" вибору:
— Низька (0.2) = модель майже завжди обирає найімовірніший токен (безпечно, передбачувано)
— Висока (2.0) = розподіл "розмазується", модель може обрати "hello worm" замість "hello world"

Після "hello world!" модель більше не має підказки — генерація стає випадковою. Спробуй натиснути ще кілька разів і побачиш хаос.

Цей самий алгоритм використовують GPT-4, Claude, Gemini — різниця лише в масштабі (1.8M vs 1,800,000M параметрів), BPE-токенізації і KV-cache для швидкості.

@torch.no_grad()
def generate(self, idx, max_new_tokens, temperature=1.0):
    for _ in range(max_new_tokens):
        idx_cond = idx[:, -BLOCK_SIZE:]              # обрізаємо до контексту
        logits, _ = self(idx_cond)                    # forward pass
        logits = logits[:, -1, :] / temperature       # остання позиція / temp
        probs = F.softmax(logits, dim=-1)             # logits → ймовірності
        nxt = torch.multinomial(probs, num_samples=1) # випадковий вибір
        idx = torch.cat([idx, nxt], dim=1)            # додаємо до тексту
    return idx
